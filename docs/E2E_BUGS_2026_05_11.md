## B-001 · 어드민 이의 제기 목록 항상 빈 화면

| 항목 | 내용 |
|---|---|
| 발견 시점 | 시나리오 2 Playwright admin 단계 |
| 영향 범위 | 어드민 `/disputes` 페이지 전체 |
| 심각도 | High — 운영자가 이의 제기를 처리할 수 없음 |
| 상태 | ✅ Fix 배포 완료 |

### 증상
어드민 좌측 메뉴 "이의 제기" 진입 시 `데이터 없음`만 표시. DB(`disputes` 테이블)에는 실제 PENDING dispute 3건이 있음.

### 원인 (2-stack 버그)

#### 1) 서버: `/admin/disputes` SQL이 삭제된 컬럼 참조
`server/src/modules/disputes/disputes.routes.ts`의 admin list 쿼리가 이전에 제거된 `m.requester_verification_code` / `m.opponent_verification_code` 컬럼을 SELECT하고 있었음. 이의 제기 시스템을 "우리 만났어요" 흐름으로 전환할 때 컬럼은 ALTER로 빠졌는데 이 쿼리는 같이 정리되지 않음.

```
ERROR: column m.requester_verification_code does not exist
```

매번 500을 받고 axios catch만 도는데, admin 페이지가 catch에서 빈 응답 처리 → 빈 테이블.

#### 2) Admin 프론트: PaginatedResponse normalize 누락
`admin/src/api/disputes.api.ts` 의 `disputesApi.list`:
```ts
return response.data.data;   // 배열만 반환
```
서버 응답은 `{ success, data: Dispute[], meta }`. 그런데 호출부 `DisputeListPage.tsx`는:
```ts
data?.items?.filter(…)   // items 키 기대
```
서버 fix만으로는 화면이 정상 동작 안 됨. API client에서 `{ items, total, page, pageSize, totalPages }` 형태로 normalize 필요.

### 수정
- `server/src/modules/disputes/disputes.routes.ts`: 잔여 컬럼 SELECT 제거 (PR 직접 반영, 무중단 배포)
- `admin/src/api/disputes.api.ts`: 응답 normalize 후 `PaginatedResponse<Dispute>` 형태로 반환
- admin 재빌드 + rsync 배포

### 검증
admin token으로 직접 `GET /admin/disputes` → `meta.total: 3` 정상 응답. Playwright `/disputes` 진입 시 PENDING 행 3건 표시.

## B-002 · `ranking_entries` 1시간마다 통째로 사라지던 문제 (이전 세션 fix, 재확인됨)

| 항목 | 내용 |
|---|---|
| 영향 범위 | 모든 핀 랭킹 (홈 화면 "순위/점수 -" 표시) |
| 심각도 | Critical |
| 상태 | ✅ Fix 배포 완료 (`worker:user_pins → pin_activities`) |

### 원인 요약
`server/src/workers/ranking-refresh.worker.ts`가 1시간마다 `user_pins` INNER JOIN으로 ranking 대상자를 필터링했는데, `user_pins` 테이블에 INSERT하는 코드 경로가 **코드베이스 전체에서 0건**이라 dead table. 따라서 worker가 매번 모든 ranking_entries를 `toDelete`로 분류해 삭제. 게임 직후엔 만들어지지만 1시간 후 사라짐.

### 수정
- worker의 3곳 `INNER JOIN u.userPins` 제거
- 신규 진입자 필터: `pin_activities` + `matches` 기반 UNION
- userCount 갱신 raw 쿼리도 `user_pins` → `pin_activities`

### 검증 (시나리오 3)
정상 결과 입력 후:
- `ranking_entries` (사당역, GOLF) 2건 등록 (테스트원·테스트투)
- Redis ZSET 동일 — 1시간 worker 후에도 보존

## B-003 · iOS 시스템 알림 권한 다이얼로그가 E2E를 막던 문제

| 항목 | 내용 |
|---|---|
| 영향 범위 | E2E 자동화 (운영 사용자 영향 없음) |
| 심각도 | Medium (테스트 인프라) |
| 상태 | ✅ TEST_MODE 빌드로 우회 |

### 증상
시뮬레이터 캡쳐가 항상 알림 권한 다이얼로그만 보임. Flutter widget tree에는 홈 화면이 mount돼 있어서 `expect`는 통과하지만 시뮬레이터 UI에선 진행 안 됨.

### 원인
- iOS 시스템 다이얼로그는 native OS UI 레이어 — Flutter `find.text('허용')` 으로 잡히지 않고 `tap` 불가
- `xcrun simctl privacy`에는 `notifications` 서비스가 없어 사전 grant 불가
- `useNativeBottomBar: true`로 native UITabBar 사용 → `find.text('매칭')` 매칭 안 됨

### 수정
- `core/push/push_notification_service.dart`: `bool.fromEnvironment('TEST_MODE')` 분기 — `requestPermission` 호출 자체 스킵
- `screens/main_tab_screen.dart`: TEST_MODE에서 `useNativeBottomBar: false`로 Flutter NavigationBar 강제

### 검증
E2E 실행 시 알림/위치 권한 다이얼로그 0건, NavigationBar 탭 텍스트 정상 매칭.

## B-004 · `_RankNumber` RangeError (`rank: 0`일 때 `colors[-1]` 접근)

| 항목 | 내용 |
|---|---|
| 영향 범위 | 랭킹 리스트 타일 (rank=0 데이터 들어올 때) |
| 심각도 | Low (사용자 노출 적음) |
| 상태 | ✅ Fix 배포 완료 |

### 원인
`app/lib/widgets/ranking/ranking_list_tile.dart`의 `_RankNumber`가 `rank <= 3` 분기에서 `colors[rank - 1]` 접근. 서버가 ranking_entries 초기화 시 `rank=0`을 주는 경우 `colors[-1]` → RangeError.

### 수정
- `if (rank >= 1 && rank <= 3)` 가드 추가
- ErrorReporter ignore 패턴도 함께 추가 (release 모드 노이즈 차단)

## B-005 · 알림 배지 한 번 켜지면 지울 방법이 없던 문제

| 항목 | 내용 |
|---|---|
| 영향 범위 | iOS 앱 아이콘 배지 |
| 심각도 | Medium — 사용자 직접 영향 |
| 상태 | ✅ Fix 배포 완료 |

### 원인
- 서버 `push.worker.ts`: 매 push마다 `unreadCount`를 `apns.aps.badge`로 전송 → 누적
- 앱에 `setApplicationIconBadgeNumber(0)` 호출 코드 0건
- 알림 읽음(`PATCH /notifications/:id/read`) 후에도 디바이스에 갱신 silent push 없음 → 영구 잔류

### 수정
- iOS AppDelegate.swift에 `kr.pins/badge` MethodChannel 추가 (`clearBadge` → `UNUserNotificationCenter.setBadgeCount(0)`, delivered 알림 제거)
- Flutter `BadgeService` wrapper → `notification_provider.markAs(All)Read` 후 호출, `main_tab_screen.didChangeAppLifecycleState(resumed)` 시 호출
- 서버 `notification.service.ts`: markRead 시점에 silent push(`content-available:1`, `badge:unreadCount`) fire-and-forget

## B-006 · 노쇼 신고 후 매치가 "채팅중" 상태로 남던 문제

| 항목 | 내용 |
|---|---|
| 영향 범위 | 노쇼 신고 흐름 (양쪽 클라이언트) |
| 심각도 | High |
| 상태 | ✅ Fix 배포 완료 |

### 원인
서버 `reportNoshow`는 트랜잭션 안에서 매치를 COMPLETED로 잘 바꿨는데, **소켓 이벤트(`MATCH_STATUS_CHANGED`) 발행이 빠짐**. 양쪽 클라이언트가 매치 상태 변경을 인지 못 해 채팅방을 그대로 유지.

### 수정
- `matching.service.ts:reportNoshow` 트랜잭션 후 `emitMatchEvent('MATCH_STATUS_CHANGED', { matchId, data: { status: 'COMPLETED' } })` 추가
- 알림 본문도 "노쇼 신고가 접수되었습니다. 매칭이 종료되었습니다."로 변경

### 추가 보강 — 허위 신고 누적 영구 정지
같은 흐름에서 `rejectNoshowReport`에 `reporterPenalty=true` 누적 카운트 추가:
- `users.false_noshow_count` 컬럼 신설
- 2회 이상이면 `User.status='SUSPENDED'` + 진행중 매치 강제 취소

## B-007 · API 에러 메시지가 토스트에 표시되지 않던 문제

| 항목 | 내용 |
|---|---|
| 영향 범위 | 매칭 요청 / 채팅 / 만남 확인 등 다수 화면 |
| 심각도 | Medium — UX (사용자가 왜 실패했는지 모름) |
| 상태 | ✅ Fix 배포 완료 |

### 원인
`create_match_screen.dart` 등에서 `if (e is DioException)`만 체크. 실제로는 `ApiClient`가 `DioException`을 `ApiException`으로 wrapping → 항상 fallback 메시지("매칭 요청에 실패했습니다.") 표시.

예: 노쇼 ban 상태에서 매칭 시도 → 서버 `MATCH_009 "노쇼 패널티로 인해..."` 응답인데 토스트는 "매칭 요청에 실패했습니다."만.

### 수정
이미 코드베이스에 있던 `extractErrorMessage(e, fallback)` 헬퍼로 통일:
- `create_match_screen.dart` (매칭 요청)
- `chat_room_screen.dart` (이미지/위치/메시지/만남 확인 4곳)
- `match_detail_screen.dart` (만남 확인)

## B-009 · 매칭/노쇼 등 핵심 알림이 `notifications`에 한 건도 안 들어가던 문제

| 항목 | 내용 |
|---|---|
| 발견 시점 | 시나리오 7 (알림/소켓) |
| 영향 범위 | **매칭/노쇼/forfeit/매치종료 등 `MatchingService` 경로 모든 알림** |
| 심각도 | **High** — 인박스 0건, unread 카운트 안 잡힘, 배지 작동 안 함 |
| 상태 | ✅ Fix 배포 완료 (`MatchingService.notificationService` lazy getter) |

### 증상
- test1/test2가 거의 동시에 `POST /matches/requests` → 자동 매칭이 성사돼 `matches.status='PENDING_ACCEPT'`까지 잘 만들어짐
- 그러나 `notifications` 테이블에 `MATCH_PENDING_ACCEPT` row 0건 (양쪽 user 모두)
- 동일 흐름에서 노쇼 신고 후 `NOSHOW_REPORT_RECEIVED` 알림도 누락

### 원인 (확정)
부팅 순서 race:
```
server.ts:263  fastify = await createApp();   // matchingRoutes 등록됨
                                              // → matching.routes.ts:22-24가
                                              //   global.__notificationService를 읽는데 아직 undefined
                                              // → new MatchingService(AppDataSource, undefined)
server.ts:268  fastify.listen()
server.ts:297  notificationService = new NotificationService(io, redis, pushQueue)
server.ts:300  global.__notificationService = notificationService  // 너무 늦음
```
이미 생성된 `MatchingService.notificationService`는 영구히 undefined → 코드의 `if (this.notificationService)` 분기가 false → `sendBulk` 호출 자체가 안 됨. 매칭/노쇼/forfeit/매치 종료 등 모든 알림이 DB·소켓·푸시 어디로도 안 감.

### 수정
`MatchingService.notificationService`를 lazy getter로 변경. 매 호출 시 `this._notificationService ?? (global as any).__notificationService` — global이 늦게 설정돼도 자동 복구.

```ts
get notificationService(): INotificationService | undefined {
  return this._notificationService ?? (global as any).__notificationService;
}
```

### 검증
시나리오 7 실행 → `notifications` 테이블에:
- `MATCH_PENDING_ACCEPT/FOUND/BOTH_ACCEPTED` 양쪽 user 각 2건
- `NOSHOW_REPORT_RECEIVED` 1건

## B-010 · 채팅 메시지 읽음 동기화 미반영 (S6에서 발견)

| 항목 | 내용 |
|---|---|
| 발견 시점 | 시나리오 6 |
| 영향 범위 | `messages.read_at` 갱신 안 됨 |
| 심각도 | Medium — unread 카운트 오작동 |
| 상태 | 🔍 재현 가능, 추적 필요 |

### 증상
- A가 3건 메시지 전송 후 B가 `POST /chat-rooms/:roomId/read` 호출 (HTTP 200)
- 그러나 `messages.read_at` 컬럼이 null 유지 → `read_from_a=0`

### 추정 원인
- 읽음 API가 별도 `chat_room_members.last_read_at`만 갱신하고 `messages.read_at`은 안 건드림
- 클라이언트는 last_read_at으로 unread 카운트 계산하지만, 서버에서 `messages.read_at` 사용하는 API가 따로 있다면 불일치 발생

### 다음 단계
- 읽음 API 엔드포인트 SQL 확인
- 채팅 unread count 계산 경로 통일
