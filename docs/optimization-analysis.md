# API/S3 요청 최적화 분석

> 분석일: 2026-04-10

## 요약

| 영역 | 현재 점수 | 주요 문제 |
|------|----------|----------|
| 앱→서버 API | 5/10 | 중복 호출, 캐싱 부재, 불필요한 폴링 |
| S3 이미지 | 4/10 | 리사이즈 없음, CDN 없음, 원본 그대로 로드 |
| 서버 DB | 5/10 | 인덱스 누락, N+1 쿼리, 과도한 데이터 반환 |

## 1. 앱→서버 API 호출 (92개 엔드포인트 분석)

### 1-1. 중복 호출 (즉시 수정 가능)

| 문제 | 영향 | 해결 |
|------|------|------|
| `/pins/nearby` — pin_repository + ranking_repository 2곳에서 호출 | 같은 요청 2번 | ranking_repository에서 제거, pin_provider 재사용 |
| 게시판 API (`/pins/:id/posts`, 댓글, 좋아요) — pin_repository + community_provider 중복 | 소스 불일치 | pin_repository로 통일 |
| `/users/me` — auth_provider에서 5번 호출 (build, 3개 로그인, refreshUser) | 앱 시작 시 불필요한 호출 | 로그인 후 1회만 호출, 이후 캐시 사용 |

### 1-2. 배칭 가능한 호출

| 현재 | 호출 수 | 개선 |
|------|---------|------|
| 매칭 요청 SENT/RECEIVED 따로 조회 | 2회 | 서버에서 한 번에 반환 |
| 팀 게시판 카테고리별 4회 동시 호출 | 4회 | 서버에서 전체 반환 → 클라이언트 필터 |

### 1-3. 불필요한 폴링

| 현재 | 빈도 | 개선 |
|------|------|------|
| 매칭 수락 대기 — 5초 간격 HTTP 폴링 (`/matches/:id/status`) | 분당 12회 | 소켓 이벤트 활용 (이미 소켓 연결 존재) |

### 1-4. 캐싱 현황

| 데이터 | 현재 캐싱 | 권장 |
|--------|----------|------|
| 핀 목록 | 버전 기반 (ETag) | 유지 |
| 매칭 목록 | TTL 5분 (Drift DB) | 유지 |
| 채팅방 목록 | TTL 30분 (Drift DB) | 유지 |
| 유저 프로필 | TTL 365일 (수동) | 유지 |
| **공지사항** | **없음** | TTL 1시간 추가 |
| **랭킹 데이터** | **없음** | TTL 10분 추가 |
| **알림 목록** | **없음** | TTL 1분 추가 |
| **팀 데이터** | **없음** | TTL 5분 추가 |
| **게임 결과** | **없음** | 로컬 DB 캐싱 추가 |

## 2. S3 이미지 요청

### 2-1. 현재 상태
- 모든 이미지에 `CachedNetworkImage` 사용 (캐싱 O)
- S3 presigned URL 업로드 (보안 O)
- 에러 핸들링/폴백 UI 적용 (UX O)

### 2-2. 문제점

| 문제 | 영향 | 해결 |
|------|------|------|
| **원본 이미지 그대로 로드** — 32px 아바타에 1MB+ 원본 사용 | 대역폭 낭비, 로딩 느림 | `cacheHeight`/`cacheWidth` 파라미터 추가 |
| **CDN 미사용** — S3 직접 URL | 글로벌 응답 느림, S3 요청 과금 | CloudFront 배포 추가 |
| **캐시 설정 없음** — DefaultCacheManager 기본값 | 캐시 무한 증가 | 최대 200MB, 30일 만료 설정 |
| **lazy loading 없음** — 게시판 이미지 즉시 전체 로드 | 불필요한 네트워크 사용 | 뷰포트 진입 시 로드 |

### 2-3. 이미지 리사이즈 권장 사이즈

| 컨텍스트 | 현재 | 권장 cacheWidth/cacheHeight |
|----------|------|---------------------------|
| 유저 아바타 (32px) | 원본 | 64x64 (2x) |
| 유저 아바타 (48px) | 원본 | 96x96 (2x) |
| 프로필 이미지 (96px) | 원본 | 192x192 (2x) |
| 팀 로고 (56px) | 원본 | 112x112 (2x) |
| 채팅 이미지 (200px) | 원본 | 400x400 (2x) |
| 게시판 그리드 | 원본 | 400x400 (2x) |

## 3. 서버 DB 비효율

### 3-1. 인덱스 누락 (가장 큰 성능 영향)

| 테이블 | 필요한 인덱스 | 영향받는 엔드포인트 |
|--------|-------------|-------------------|
| notifications | `(user_id, is_read, created_at)` | GET /notifications |
| match_requests | `(requester_id, status, expires_at)` | 매칭 큐 워커 |
| matches | `(status, created_at)` | GET /matches |
| posts | `(pin_id, is_deleted, created_at)` | GET /pins/:id/posts |

### 3-2. N+1 쿼리

| 엔드포인트 | 현재 쿼리 수 | 문제 | 해결 |
|-----------|------------|------|------|
| GET /matches/:id | 5회 | match + matchRequest + pin + encounterCount + acceptances 각각 조회 | JOIN으로 1~2회로 통합 |
| GET /notifications | 2회 | 목록 + unread count 별도 | window function으로 통합 |

### 3-3. 과도한 데이터 반환

| 엔드포인트 | 문제 | 해결 |
|-----------|------|------|
| GET /matches | User 전체 필드 (email, 비밀번호 해시 등) JOIN | `SELECT id, nickname, profileImageUrl`만 |
| GET /pins/all | 전체 핀 무제한 반환 | 이미 버전 기반이라 OK, 단 LIMIT 추가 권장 |

## 4. 우선순위별 액션 플랜

### P0 — 즉시 (30분 이내, 큰 효과)

1. **DB 인덱스 추가** — 엔티티에 `@Index()` 데코레이터 (4개 테이블)
2. **알림 COUNT 쿼리 통합** — 별도 unread count 제거
3. **이미지 cacheWidth/cacheHeight 추가** — UserAvatar, 채팅 이미지 등

### P1 — 이번 주 (중간 효과)

4. **중복 repository 호출 제거** — pins/nearby, 게시판 API 통일
5. **매칭 수락 폴링 → 소켓 전환** — 5초 폴링 제거
6. **GET /matches/:id 쿼리 통합** — 5회 → 2회
7. **매칭 요청 SENT/RECEIVED 배칭** — 2회 → 1회

### P2 — 다음 주 (장기 효과)

8. **CloudFront CDN 설정** — S3 앞에 배포
9. **이미지 캐시 매니저 설정** — 200MB 제한, 30일 만료
10. **공지/랭킹/팀 데이터 TTL 캐싱 추가**
11. **팀 게시판 4회 호출 → 1회 통합**

### P3 — 월간

12. **서버 응답 필드 최소화** — User SELECT 최적화
13. **이미지 lazy loading** — 게시판 그리드
14. **서버 사이드 이미지 리사이즈** — 업로드 시 썸네일 자동 생성 (Lambda)
